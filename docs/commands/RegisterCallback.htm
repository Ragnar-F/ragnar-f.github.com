RegisterCallback() [v1.0.47+]

Erstellt eine Maschinencode-Adresse, die, wenn sie aufgerufen wird, den Aufruf an eine Funktion im Script weiterleitet.

Adresse := RegisterCallback("Funktionsname" [, Optionen = "", Parameteranzahl = Formalanzahl, Ereignisinfo = Adresse])

Parameter

Adresse

Bei Erfolg gibt RegisterCallback() eine numerische Adresse zurück, die mit DllCall() oder Ähnlichem aufgerufen werden kann. Bei Misserfolg wird ein leerer String zurückgegeben. Fehler treten auf, wenn Funktionsname: 1) nicht existiert; 2) zu viele oder zu wenig Parameter in Bezug auf Parameteranzahl akzeptiert; oder 3) einen beliebigen ByRef-Parameter akzeptiert.

Funktionsname

Der Name einer Funktion, der in Anführungszeichen gesetzt werden muss, falls dieser ein literaler String ist. Diese Funktion wird automatisch aufgerufen, sobald die Adresse aufgerufen wird. Die Funktion ermittelt auch die Parameter, die an die Adresse übergeben wurden.

[v1.1.06+]: Anstelle eines Funktionsnamens kann ein Funktionsverweis übergeben werden.

Optionen

Es können beliebig viele der folgenden Wörter angegeben werden. Die Optionen werden mit einem Leerzeichen getrennt (z.B. C Fast).

Fast oder F: Verhindert, dass jedes Mal ein neuer Thread gestartet wird, wenn der Funktionsname aufgerufen wird. Obwohl das zu besseren Ergebnissen führt, sollte das verhindert werden, sobald der aufgerufene Thread von der Adresse variiert (wenn der Callback beispielsweise von einer eingehenden Nachricht ausgelöst wird). Denn Funktionsname wird in der Lage sein, globale Einstellungen wie ErrorLevel, A_LastError und das zuletzt gefundene Fenster für jeden Thread, der beim Aufruf gestartet wird, zu ändern. Weitere Informationen können bei den Bemerkungen gefunden werden.

CDecl oder C: Passt die Adresse an die "C"-Aufrufkonvention an. Diese kann normalerweise weggelassen werden, da die Standard-Aufrufkonvention für Callbacks viel gebräuchlicher ist.

Parameteranzahl

Die Anzahl der Parameter, die dem Aufrufer von der Adresse übergeben werden. Ist dieser Parameter nicht vorhanden, wird standardmäßig die Anzahl der Pflichtparameter in der Definition von Funktionsname verwendet. Auf jeden Fall sollte sichergestellt werden, dass der Aufrufer genau diese Anzahl von Parametern übergibt.

EventInfo

Ein Integer, den Funktionsname in A_EventInfo sieht, sobald Funktionsname über diese Adresse aufgerufen wird. Das ist nützlich, wenn Funktionsname von mehr als einer Adresse aufgerufen wird. Ist dieser Parameter nicht vorhanden, gilt standardmäßig die Adresse. Hinweis: Im Gegensatz zu anderen globalen Einstellungen wird A_EventInfo des aktuellen Threads nicht vom schnellen Modus beeinflusst.

Wenn die EXE-Datei, die das Script ausführt, im 32-Bit-Format ist, muss dieser Parameter ein Wert zwischen 0 und 4294967295 sein. Falls die EXE-Datei im 64-Bit-Format ist, kann dieser Parameter ein 64-Bit-Integer sein. Auch wenn A_EventInfo für gewöhnlich einen vorzeichenlosen Integer zurückgibt, unterstützt AutoHotkey vorzeichenlose 64-Bit-Integer nicht im vollen Umfang, daher bewirken einige Operationen, dass der Wert in den vorzeichenbehafteten Bereich rutscht.

Die Parameter der Callback-Funktion

Eine Funktion, die einer Callback-Adresse zugewiesen ist, kann bis zu 31 Parameter akzeptieren. Optionale Parameter sind erlaubt und nützlich, wenn die Funktion von mehreren Aufrufern aufgerufen wird.

32-Bit: Alle eingehenden Parameter sind vorzeichenlose 32-Bit-Integer. Wenn ein eingehender Parameter als vorzeichenbehafteter Integer vorgesehen ist, können negative Zahlen mit einem der folgenden Beispielen offenbart werden:

; Methode #1
If wParam > 0x7FFFFFFF
    wParam := -(~wParam) - 1

; Methode #2: Basiert auf die Tatsache, dass AutoHotkey von Haus aus vorzeichenbehaftete 64-Bit-Integers verwendet.
wParam := wParam << 32 >> 32

64-Bit: Alle eingehenden Parameter sind vorzeichenbehaftete 64-Bit-Integer. AutoHotkey unterstützt von Haus aus keine vorzeichenlose 64-Bit-Integers.

Wenn ein eingehender Parameter vom Aufrufer als String vorgesehen ist, ist das, was aktuell empfangen wird, die aktuelle Adresse des Strings. Mit StrGet kann der String selbst abgerufen werden:

MeinString := StrGet(MeinParameter)  ; AHK_L 46+

Wenn ein eingehender Parameter die Adresse einer Struktur ist, können einzelne Elemente daraus extrahiert werden, wenn die Schritte in "DllCall-Strukturen" befolgt werden.

[AHK_L 60+]: Ist die Funktion als variadisch deklariert, wird dem letzte Parameter die Adresse des ersten Callback-Parameters zugewiesen, die nicht einem Script-Parameter zugewiesen worden ist. Zum Beispiel:

callback := RegisterCallback("Func", "F", 1)  ; Parameteranzahl muss angegeben werden.
Func("Funktion wurde direkt aufgerufen.")  ; Func direkt aufrufen.
DllCall(callback, float, 10.5)  ; Func über Callback aufrufen.
Func(params*) {
    If IsObject(params)
        MsgBox % params[1]
    Else
        MsgBox % NumGet(params+0, 0, "float")
}

Die meisten Callbacks verwenden die stdcall-Aufrufkonvention, die eine feste Anzahl an Parametern voraussetzt. In diesem Fall muss Parameteranzahl beim Erstellen des Callbacks angegeben werden. Die Parameteranzahl kann in 64-Bit-Builds und bei Cdecl-Callbacks in 32-Bit-Builds weggelassen werden. In solchen Fällen erhalten alle optionalen Parameter ihre Standardwerte und werden für die Adressenberechnung ignoriert, um in params gespeichert werden zu können.

Was die Funktion per Return zurückgeben sollte

Wenn die Funktion ein Return ohne Parameter verwendet oder ein leerer Wert wie "" angegeben ist (oder überhaupt nicht Return verwendet), wird dem Aufrufer von Adresse eine 0 zurückgegeben. Ansonsten sollte die Funktion einen Integer zwischen -2147483648 und 4294967295 zurückgeben, der dann dem Aufrufer von Adresse zurückgegeben wird.

Fast vs. Slow

Der voreingestellte/langsame Modus bewirkt, dass die Funktion zu Beginn mit den Standardwerten von Einstellungen wie SendMode und DetectHiddenWindows gestartet wird. Diese können im automatischen Ausführungsbereich geändert werden.

Der schnelle Modus hingegen erbt die globalen Einstellungen von einem beliebigen Thread, der zum Zeitpunkt des Funktionsaufrufs läuft. Desweiteren werden die von der Funktion vorgenommenen Änderungen an den globalen Einstellungen (einschließlich ErrorLevel und das zuletzt gefundene Fenster) auch für den aktuellen Thread wirksam. Folglich sollte der schnelle Modus nur verwendet werden, wenn genau bekannt ist, welche Threads von der Funktion beeinflusst werden.

Damit ein Callback weder von sich selbst noch von einem anderen Thread unterbrochen wird, kann in der erste Zeile der Funktion Critical angegeben werden. Allerdings hilft das nicht viel, wenn die Funktion durch eine eingehende Nachricht kleiner als 0x312 indirekt aufgerufen wird (die Erhöhung des Intervalls von Critical könnte helfen). Desweiteren kann Critical die Funktion nicht daran zu hindern, etwas zu tun, das zum indirekten Selbstaufruf führen könnte, wie z. B. durch SendMessage oder DllCall.

Speicher

Jedes mal, wenn RegisterCallback() verwendet wird, wird eine kleine Speichermenge (32 Bytes plus System-Overhead) reserviert. Da das Betriebssystem diesen Speicher automatisch freigibt, sobald das Scripts beendet wird, muss jedes Script, das eine kleine feste Anzahl an Callbacks reserviert, nicht unbedingt den Speicher freigegeben. Hingegen sollte ein Script, das RegisterCallback() unendlich oft aufruft, folgendes direkt bei unbenutzten Callbacks aufrufen:

DllCall("GlobalFree", "Ptr", Adresse, "Ptr")

Siehe auch

DllCall(), OnMessage(), OnExit, OnClipboardChange, Sort's Callback, Critical, Post/SendMessage, Funktionen, Liste mit Fensternachrichten, Threads

Beispiele

; Beispiel: Das folgende Script zeigt eine Zusammenfassung aller Fenster auf höchster Ebene an.

; Aus Performance- und Speichergründen wird RegisterCallback() nur einmal für einen bestimmten Callback aufgerufen:
If not EnumAdresse  ; Schneller Modus ist okay, weil er nur vom diesen Thread aufgerufen wird:
    EnumAdresse := RegisterCallback("EnumWindowsProc", "Fast")

DetectHiddenWindows On  ; Durch den schnellen Modus wird diese Einstellung auch für Callback wirksam.

; Kontrolle an EnumWindows() übergeben, die Callback wiederholend aufruft:
DllCall("EnumWindows", Ptr, EnumAdresse, Ptr, 0)
MsgBox %Ausgabe%  ; Zeigt die gesammelten Informationen von Callback an.

EnumWindowsProc(hwnd, lParam)
{
    global Ausgabe
    WinGetTitle, Titel, ahk_id %hwnd%
    WinGetClass, Klasse, ahk_id %hwnd%
    If Titel
        Ausgabe .= "HWND: " . hwnd . "`tTitel: " . Titel . "`tKlasse: " . Klasse . "`n"
    Return true  ; Setzt EnumWindows() fort, bis alle Fenster abgearbeitet wurden.
}

 

; Beispiel: Das folgende Script zeigt, wie ein GUI-Fenster zu einer Unterklasse gemacht wird,
; indem dessen WindowProc an einen neuen WindowProc weitergeleitet wird. In diesem Fall wird die Hintergrundfarbe
; des Text-Steuerelements auf eine benutzerdefinierte Farbe gesetzt.

Texthintergrundfarbe := 0xFFBBBB  ; Eine benutzerdefinierte Farbe im BGR-Format.
Pinsel := DllCall("CreateSolidBrush", UInt, Texthintergrundfarbe)

Gui, Add, Text, HwndMeinTextHwnd, Hier ein Text mit einer`nbenutzerdefinierter Hintergrundfarbe.

GuiHwnd := WinExist()

; 64-Bit-Scripts müssen SetWindowLongPtr statt SetWindowLong aufrufen:
SetWindowLong := A_PtrSize=8 ? "SetWindowLongPtr" : "SetWindowLong"

WindowProcNeu := RegisterCallback("WindowProc", ""  ; Mit "" wird der schnelle Modus beim Setzen der Unterklasse verhindert.
    , 4, MeinTextHwnd)  ; Parameteranzahl muss genau angegeben werden, wenn der EventInfo-Parameter vorhanden ist.
WindowProcAlt := DllCall(SetWindowLong, Ptr, GuiHwnd, Int, -4  ; -4 ist GWL_WNDPROC
    , Ptr, WindowProcNeu, Ptr)  ; Wert muss auf Ptr oder UPtr anstatt auf Int gesetzt werden.

Gui Show
Return

WindowProc(hwnd, uMsg, wParam, lParam)
{
    Critical
    global Texthintergrundfarbe, Pinsel, WindowProcAlt
    If (uMsg = 0x138 && lParam = A_EventInfo)  ; 0x138 ist WM_CTLCOLORSTATIC.
    {
        DllCall("SetBkColor", UInt, wParam, Ptr, Texthintergrundfarbe)
        Return Pinsel  ; Gibt HBRUSH zurück, um dem OS mitzuteilen, dass wir HDC geändert haben.
    }
    ; Ansonsten (da oben nichts zurückgegeben wurde) werden alle unbehandelten Ereignisse an das originale WindowProc übergeben.
    Return DllCall("CallWindowProc", Ptr, WindowProcAlt, Ptr, hwnd, UInt, uMsg, Ptr, wParam, Ptr, lParam)
}
