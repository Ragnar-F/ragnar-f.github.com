SetKeyDelay

Bestimmt die Verzögerung, die nach jeder Tastatureingabe durch Send und ControlSend auftreten wird.

SetKeyDelay [, Verzögerung, Druckdauer, Play]

Parameter

Verzögerung

Zeit in Millisekunden (kann ein Ausdruck sein). Verwende -1, um die Verzögerung vollständig zu deaktivieren, oder 0, um die kleinst mögliche Verzögerung zu veranlassen (sobald jedoch der Play-Parameter vorhanden ist, erzeugen sowohl 0 als auch -1 keine Verzögerung). Lasst diesen Parameter leer, um die aktuelle Verzögerung beizubehalten.

Wenn SetKeyDelay nie im Script benutzt wird, dann wird für die Verzögerung des traditionellen SendEvent-Modus standardmäßig 10 verwendet. Beim SendPlay-Modus verwendet die Verzögerung standardmäßig -1. Die Druckdauer ist standardmäßig bei beiden Modi -1.

Druckdauer

Bestimmte Spiele und andere spezialisierte Anwendungen benötigen möglicherweise eine Verzögerung innerhalb jeder Tastatureingabe, d. h. nach dem Drücken der Taste, aber vor dem Loslassen.

Verwendet -1 für keine Verzögerung (Standard) und 0 für die kleinst mögliche Verzögerung (sobald jedoch der Play-Parameter vorhanden ist, erzeugen sowohl 0 als auch -1 keine Verzögerung). Wenn dieser Parameter weggelassen wird, dann bleibt die aktuelle Druckdauer unverändert.

Hinweis: Die Druckdauer erzeugt zudem eine Verzögerung nach jeder Zustandsänderung eines Modifikators (STRG, ALT, UMSCHALT und WIN) und wird benötigt, damit die Tasten gesendet werden.

Dieser Parameter kann ein Ausdruck sein.

Play
[v1.0.43+]

Das Wort Play gilt für die oben genannten Einstellungen im SendPlay-Modus, jedoch nicht für den traditionellen SendEvent-Modus. Wenn ein Script diesen Parameter nie verwendet, dann ist die SendPlay-Verzögerung immer -1/-1.

Bemerkungen

Hinweis: SetKeyDelay wird nicht von SendInput berücksichtigt: in diesem Modus gibt es keine Verzögerung zwischen den Tastatureingaben. Das Gleiche gilt für Send, wenn SendMode Input aktiv ist.

Eine kurze Verzögerung (Ruhephase) erfolgt automatisch nach jeder Tastatureingabe mit Send oder ControlSend. Das geschieht, um die Zuverlässigkeit der Scripts zu verbessern, da manchmal ein Fenster nicht mit einer Flut von Tastatureingaben mithalten kann.

Durch die Granularität (Ungenauigkeit) der Zeiterfassung im Betriebssystem werden Verzögerungen möglicherweise auf das nächstmögliche Vielfache von 10 oder 15 aufgerundet. Zum Beispiel ist eine Verzögerung zwischen 1 und 10 in den meisten XP-Systemen (und wahrscheinlich auch in 2000) gleichbedeutend mit 10 oder 15.

Beim Send/SendEvent-Modus führt eine Verzögerung von 0 Sleep(0) intern aus, die den Rest der Script-Zeitscheibe für einen anderen Prozess zurückgibt. Besteht kein Bedarf, wird Sleep(0) keine Verzögerung erzeugen. -1 würde hingegen nie eine Verzögerung erzeugen. Für eine bessere Zuverlässigkeit ist 0 als eine Alternative zu -1 empfehlenswert.

Wenn die Verzögerung auf -1 gesetzt wird, dann spielt die Prozess-Priorität eine große Rolle, weil sie bestimmt, wie schnell Tastatureingaben gesendet werden können, wenn der traditionelle SendEvent-Modus verwendet wird. Die Priorität des Scripts kann mit Process, Priority,, High erhöht werden. Obwohl dadurch die Tastatureingaben schneller gesendet werden, als vom aktiven Fenster verarbeitet, werden sie automatisch vom System zwischengespeichert. Zwischengespeicherte Tastatureingaben werden weiterhin an das Zielfenster gesendet, auch nachdem das Ende des Send-Befehls erreicht wurde (selbst wenn das Fenster nicht länger aktiv ist). Das ist für gewöhnlich harmlos, denn jede nachfolgende gesendete Tastatureingabe ans selbe Fenster wird in der Warteschlange für zwischengespeicherte Tastatureingaben ganz nach hinten gesetzt.

Die integrierte Variable A_KeyDelay enthält die aktuelle Einstellung der Verzögerung im Send/SendEvent-Modus. Es gibt keine integrierte Variable für die Druckdauer oder Verzögerung von SendPlay.

Jeder neu gestartete Thread (z. B. ein Hotkey, ein benutzerdefinierter Menüpunkt oder eine zeitgesteuerte Subroutine) beginnt mit der Standardeinstellung des Befehls. Diese Standardeinstellung kann man ändern, wenn dieser Befehl bereits im automatischen Ausführungsbereich (am Anfang des Scripts) definiert wird.
